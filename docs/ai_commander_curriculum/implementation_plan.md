# プロジェクト資産査定と「AI実装指揮官」カリキュラム実行計画

## 1. 現在の資産価値のシビアな査定 (2026年3月現在)

### 【現状の売却価値】
**推定算定額:** 1,200万円 〜 1,800万円（マイクロSaaS市場基準）
- **ポジティブ要因:** 2ヶ月という短期間でNext.js, TypeScript, Supabase, Sentryを用いた最新スタックのSPAを構築できている点。Playwright/Jestの基盤がある点。
- **ネガティブ要因:** AI生成（バイブコーディング）の特性上、過剰なコンポーネント分割、不要な再レンダリング、`test_output_*.txt` に見られるような場当たり的なテスト運用による保守性の懸念。

### 【改善後の推定価値】
**推定算定額:** 6,000万円 〜 9,000万円（プレミアム価値 / 5倍）
- 1ヶ月のカリキュラム完遂後、コードは「完全なテストカバレッジ」「自律的拡張性」「エンタープライズ水準の堅牢性」を持つことになり、属人性が排除され買い手にとって理想的なクリーン・アセットになります。

### 【技術的負債の特定】
- **AI特有の冗長コード:** `components/` 内における類似コンポーネントの乱立、不要な`useEffect`の連鎖、過剰な型定義（`types/`と`lib/`の重複可能性）。
- **セキュリティ・アーキテクチャ・リスク:** `verify_rls.sql` や `simulation_data.sql` などのテスト/DB検証スクリプトがルート直下に混在。本番環境におけるRLS（Row Level Security）の抜け漏れや、環境変数の誤露出リスク（Sentry設定等）。
- **依存関係の肥大化:** `package.json` における、未使用パッケージの残留やPnP環境下でのモジュール解決パスの複雑化によるビルド速度の低下。

## 2. 価値を5倍にする「バイブエンジニアリング」カリキュラムの実行計画

### Week 1【仕様工学と自律運用基盤】: SKILL定義と構造化PRDからのワンショット実装フロー
- **概要:** 曖昧な自然言語（バイブ）から脱却し、これ以上技術的負債を増やさないための「プロジェクト独自のAIコーディングルール（`SKILL.md`）」と「構造化PRD（`PRD_TEMPLATE.md`）」を定義・適用する。
- **タスク:**
  1. プロジェクト固有の実装方針・ルールを規定した `.agent/skills/SKILL.md` の作成とレビュー。
  2. プロジェクトルート `docs/` 等への `PRD_TEMPLATE.md` の作成。
  3. エージェントがPRDを読み込み、DBスキーマ変更 -> API -> UIコンポーネント までを誤差なく一貫生成するパターンの検証と固定化。

### Week 2【信頼性と継続的リファクタリング】: 適応型TDDとテスト網羅率80%超え
- **概要:** Week 1で定義したルールに基づき、エージェントにテストファーストの思想（適応型TDD）を注入。既存の「不完全なコード」を保護しつつ、テスト網羅率を飛躍的に高めてからリファクタリングを回す。
- **タスク:**
  1. 既存の `test_output_*.txt` 等の整理と、Jest/Playwright環境の最適化（裏プロセスとの連携）。
  2. 主要なビジネスロジック（`lib/`）およびコアUIに対するユニットテスト・統合テストの実装。
  3. テストが通る状態を維持したまま、冗長なコンポーネントや不要なレンダリングの段階的リファクタリング。

### Week 3【価値向上とUIリフレッシュ】: Google Stitch連携とモダンデザイン化
- **概要:** テストと整った生成基盤の上で、Google Stitchと連携し、Radix UI資産をベースとした最新のモダンWebデザインへの全局面リフレッシュや機能拡張を行う。
- **タスク:**
  1. Google Stitchと連携したデザイン・トークンの整理と `tailwind.config` / `components.json` の最適化。
  2. アプリ全体のUIリフレッシュと動的アニメーション（バイブス）の注入。
  3. 新機能（例: 管理者ダッシュボード拡張等）の安全で高速なデモ実装。

### Week 4【市場価値化】: 資産証跡の整理とポートフォリオ化
- **概要:** バイブコーディングによるブラックボックスを解消し、誰が見ても（特に買収先が）理解できる「設計ドキュメント・テスト結果・監査録画」へと変換。
- **タスク:**
  1. 生成されたドキュメント（`implementation_plan.md`, `walkthrough.md` 等）の最終監査と体系化。
  2. Playwrightによるエンドツーエンドのテスト実行録画（エージェントによる自動生成）。
  3. アーキテクチャ図（Mermaid）の生成と、SaaS売却用テクニカル・デューデリジェンス用資料の完成。

## 3. マシンパワー最適化指示（M5 MacBook Air 24GB向け）

### エージェント3体並列稼働アーキテクチャ
N1チップ＋24GB RAM＋超高速SSDの性能をフル活用するため、以下のターミナル/プロセス分割でAGを運用します。

1. **Agent A (Dev - 開発・生成専任):**
   - **役割:** PRDに基づくコードの記述、リファクタリング、UIコンポーネント生成。
   - **リソース配分:** 最も計算リソースを消費。メインのIDEコンテキストに常駐。
2. **Agent B (Test - 検証・品質保証専任):**
   - **役割:** Agent Aが書いたディレクトリを監視し、`jest --watch` および `playwright test` をバックグラウンドで自動実行。カバレッジレポートの常時更新。
   - **リソース配分:** SSDの高速I/Oを活かし、メモリはテストランナー用に固定（4-6GB程度）。
3. **Agent C (Monitor - 監視・静的解析専任):**
   - **役割:** リアルタイムでの `tsc --noEmit`、`eslint`、およびSentryビルド監視の実行。型エラーや不要なインポートを瞬時に検知し、Agent Aへフィードバック。

### コンテキスト管理のルール
- **コンテキストの分離:** 3つのエージェントが同一ファイルを同時に書き換える競合（Gitコンフリクト）を防ぐため、Agent BとCは **「読み取り/実行専用(Read-only)」** モードで稼働させ、修正提案のみをAgent Aのコンテキスト（Task list）に渡すようにします。
- **SSD最適化:** Yarn PnP（Plug'n'Play）のキャッシュとNext.jsのビルドキャッシュ (`.next`) は超高速SSDの恩恵を最大化します。I/Oボトルネックはないため、並列での型チェックとLint実行を躊躇なく回し続けてください。
